# Ex16 — Integração final

## Contexto

Agora tudo deve funcionar junto: classificação, roteamento, consulta MCP e resposta final.

Na aula, este fluxo pode ser demonstrado com o cliente didático em memoria ou com um cliente real apontando para o servidor `FastMCP`. A lógica do agente deve ser a mesma nos dois casos.

## Arquivos permitidos

```text
stack_sentinel/agent/graph.py
stack_sentinel/agent/nodes.py
```

## Contrato

Implemente:

```python
run_stack_sentinel_flow(
    state: AgentState,
    llm: LLMClient,
    mcp_client: MCPClient,
) -> AgentState
```

Fluxo esperado:

1. classificar intenção;
2. escolher rota com `route_by_intent`;
3. executar o node correspondente;
4. executar `final_answer_node`;
5. retornar o state final.

Rotas:

- ticket -> `fetch_ticket_node`;
- build -> `fetch_build_node`;
- docs -> `fetch_docs_node`;
- fallback -> `fallback_node` e/ou resposta final de fallback.

## Validação

```bash
python run.py test ex16
python run.py demo
```

## Desafios opcionais

- Adicione uma execução que acumule `steps` no state para mostrar a trilha completa do agente, incluindo classificação, rota escolhida, chamada MCP e geração da resposta final. Essa trilha deve ser útil para explicar o fluxo em sala.
- Crie uma versão assíncrona do fluxo, preparando o código para clientes MCP remotos. O desafio é preservar a mesma ordem lógica do fluxo síncrono enquanto permite `await` nas chamadas que poderiam depender de rede.
