# Ex16 — Integração final

## Contexto

Agora tudo deve funcionar junto: classificação, roteamento, consulta MCP e resposta final. A implementação pode continuar expondo `run_stack_sentinel_flow(...)` como função de conveniência, mas a organização deve refletir o modelo de LangGraph: nodes pequenos, state compartilhado e roteamento explícito.

Na aula, este fluxo pode ser demonstrado com um cliente MCP de teste apontando para o servidor `FastMCP` fake, ou com um cliente real apontando para o servidor `FastMCP`. A lógica do agente deve ser a mesma nos dois casos.

## Arquivos permitidos

```text
stack_sentinel/agent/graph.py
stack_sentinel/agent/nodes.py
```

## Contrato

Implemente:

```python
run_stack_sentinel_flow(
    state: AgentState,
    llm: LLMClient,
    mcp_client: MCPClient,
) -> AgentState
```

Fluxo esperado:

1. classificar intenção;
2. escolher rota com `route_by_intent`;
3. executar o node correspondente;
4. executar `final_answer_node`;
5. retornar o state final.

Ao montar uma versão com `StateGraph`, use nomes de nodes compatíveis com as rotas de `route_by_intent`.

Rotas:

- ticket -> `fetch_ticket_node`;
- build -> `fetch_build_node`;
- docs -> `fetch_docs_node`;
- fallback -> `fallback_node` e/ou resposta final de fallback.

## Validação

```bash
python run.py test ex16
python run.py demo
```

## Desafios opcionais

- Adicione uma execução que acumule `steps` no state para mostrar a trilha completa do agente, incluindo classificação, rota escolhida, chamada MCP e geração da resposta final. Essa trilha deve ser útil para explicar o fluxo em sala.
- Crie uma função `compile_stack_sentinel_graph(...)` que monte o fluxo final com `StateGraph` e `add_conditional_edges`. O desafio é injetar `llm` e `mcp_client` nos nodes sem colocá-los dentro do state.
