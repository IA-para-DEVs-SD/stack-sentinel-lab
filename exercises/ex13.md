# Ex13 — Node que consulta build

## Contexto

O fluxo de build segue o mesmo padrão do fluxo de ticket, mas usando `build_id` e a tool de build.

Essa tool foi exposta no `FastMCP` no Ex05, então o node deve continuar falando com MCP via cliente, sem conhecer o `MockServiceClient`.

## Arquivo permitido

```text
stack_sentinel/agent/nodes.py
```

## Contrato

Implemente:

```python
fetch_build_node(state: AgentState, mcp_client: MCPClient) -> AgentState
```

Regras:

- use `state["build_id"]` se ele já existir;
- se não existir, extraia o ID de `state["user_input"]` usando o padrão `BLD-000`;
- chame `mcp_client.call_tool(BUILD_TOOL_NAME, {"build_id": build_id})`;
- quando `ok=True`, salve o resultado em `context`, preencha `build_id` e deixe `error` como `None`;
- quando não houver ID ou a tool falhar, preencha `error`.

## Validação

```bash
python run.py test ex13
```

## Desafios opcionais

- Quando houver `ticket_id` no state e a resposta do build trouxer serviço, combine o contexto existente com o contexto do build em uma visão única. O objetivo é treinar composição de evidências sem perder o resultado anterior.
- Classifique builds em categorias de causa provável a partir de `failed_step` e `log_excerpt`, como `test_failure`, `timeout`, `dependency` ou `unknown`. Essa classificação deve ser heurística e explícita, não uma nova chamada de LLM.
