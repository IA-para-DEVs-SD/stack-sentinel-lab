# Ex12 — Node que consulta ticket

## Contexto

Quando a rota for ticket, o agente deve consultar uma tool MCP e salvar o resultado no state.

Nos testes, `MCPClient` ainda chama um servidor em memoria. Na aplicacao real, esse mesmo node representa a fronteira onde entraria um cliente conectado ao servidor `FastMCP`.

## Arquivo permitido

```text
stack_sentinel/agent/nodes.py
```

## Contrato

Implemente:

```python
fetch_ticket_node(state: AgentState, mcp_client: MCPClient) -> AgentState
```

Regras:

- use `state["ticket_id"]` se ele já existir;
- se não existir, extraia o ID de `state["user_input"]` usando o padrão `TCK-000`;
- chame `mcp_client.call_tool(TICKET_TOOL_NAME, {"ticket_id": ticket_id})`;
- quando `ok=True`, salve o resultado em `context`, preencha `ticket_id` e deixe `error` como `None`;
- quando não houver ID ou a tool falhar, preencha `error` e não quebre a execução.

## Validação

```bash
python run.py test ex12
```

## Desafios opcionais

- Faça o node armazenar metadados da chamada MCP junto com o contexto, como nome da tool, argumentos enviados e talvez um status da chamada. Isso ajuda a depurar o agente sem precisar inspecionar logs externos.
- Implemente uma política simples de retry para falhas transitórias da tool. O retry não deve acontecer quando o problema for entrada inválida, como ausência de `ticket_id`, porque repetir a mesma chamada não resolveria a causa.
