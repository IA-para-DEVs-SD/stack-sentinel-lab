# Ex05 — Criar segunda tool

## Contexto

O agente também precisará investigar builds quebrados. Esta tool segue o mesmo padrão da tool de ticket e deve ficar disponível também no servidor `FastMCP` criado no Ex04.5.

## Arquivos permitidos

```text
stack_sentinel/mcp_server/tools.py
stack_sentinel/mcp_server/fastmcp_server.py
stack_sentinel/mcp_server/server.py
```

## Contrato

Implemente:

```python
fetch_build_status(build_id: str, client: Optional[MockServiceClient] = None) -> dict
```

Em sucesso, retorne:

```python
{
    "ok": True,
    "id": "BLD-203",
    "status": "failed",
    "service": "auth-service",
    "branch": "main",
    "failed_step": "integration-tests",
    "log_excerpt": "..."
}
```

Em falha, retorne `ok=False` e `error`.

Registre a tool `fetch_build_status` no servidor `FastMCP`, delegando para `tools.fetch_build_status`.

O registro didático em `server.py` deve continuar funcionando para os testes locais.

## Validação

```bash
python run.py test ex05
```

## Desafios opcionais

- Organize o registro das tools em uma função auxiliar reutilizável, por exemplo `register_tools(mcp)`. O objetivo é preparar o servidor para crescer com novas tools sem transformar `create_fastmcp_server()` em uma lista longa de decorators.
- Adicione validação explícita no handler FastMCP de build para lidar com `build_id` vazio ou inválido. A resposta deve ser controlada e fácil de diagnosticar, sem deixar exceções internas vazarem para o cliente MCP.
