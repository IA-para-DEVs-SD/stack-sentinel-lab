# Ex05 — Solução

Arquivo:

```text
stack_sentinel/mcp_server/tools.py
```

Depois do Ex04.5, a regra de dominio continua em `tools.py`, mas a capacidade tambem deve ser exposta no `FastMCP`.

Substitua a função `fetch_build_status` por:

```python
def fetch_build_status(build_id: str, client: Optional[MockServiceClient] = None) -> Dict[str, Any]:
    if not build_id:
        return {"ok": False, "error": "build_id is required"}

    client = client or MockServiceClient()
    response = client.get_build(build_id)

    if not response.get("ok"):
        return {"ok": False, "error": response.get("error", "build not found")}

    data = response.get("data") or {}
    return {
        "ok": True,
        "id": data.get("id"),
        "status": data.get("status"),
        "service": data.get("service"),
        "branch": data.get("branch"),
        "failed_step": data.get("failed_step"),
        "log_excerpt": data.get("log_excerpt"),
    }
```

A função `register_build_tool` já está implementada no scaffolding e deve continuar como está para os testes locais.

No arquivo `stack_sentinel/mcp_server/fastmcp_server.py`, adicione a segunda tool:

```python
from stack_sentinel.mcp_server import tools


def create_fastmcp_server():
    from mcp.server.fastmcp import FastMCP

    mcp = FastMCP("stack-sentinel-mcp")

    @mcp.tool()
    def fetch_ticket_context(ticket_id: str) -> dict:
        """Busca contexto normalizado de um ticket."""
        return tools.fetch_ticket_context(ticket_id)

    @mcp.tool()
    def fetch_build_status(build_id: str) -> dict:
        """Busca status e evidencias de um build."""
        return tools.fetch_build_status(build_id)

    return mcp
```

## Solução alternativa válida

Também é aceitável chamar diretamente o endpoint FastAPI `GET /builds/{build_id}` via `client.get_json(...)`, desde que o retorno final continue normalizado.

```python
def fetch_build_status(build_id: str, client: Optional[MockServiceClient] = None) -> Dict[str, Any]:
    if not build_id:
        return {"ok": False, "error": "build_id is required"}

    client = client or MockServiceClient()
    response = client.get_json(MOCK_ENDPOINTS["builds"].format(build_id=build_id))

    if not response.get("ok"):
        return {"ok": False, "error": response.get("error", "build not found")}

    data = response.get("data") or {}
    return {
        "ok": True,
        "id": data.get("id"),
        "status": data.get("status"),
        "service": data.get("service"),
        "branch": data.get("branch"),
        "failed_step": data.get("failed_step"),
        "log_excerpt": data.get("log_excerpt"),
    }
```

Para essa alternativa, garanta que `MOCK_ENDPOINTS` esteja importado no arquivo:

```python
from stack_sentinel.clients.mock_service_client import MOCK_ENDPOINTS, MockServiceClient
```

## Desafios opcionais — solucao proposta

Uma forma simples de organizar o crescimento do arquivo é separar o registro:

```python
def register_tools(mcp):
    @mcp.tool()
    def fetch_ticket_context(ticket_id: str) -> dict:
        return tools.fetch_ticket_context(ticket_id)

    @mcp.tool()
    def fetch_build_status(build_id: str) -> dict:
        if not build_id:
            return {"ok": False, "error": "build_id is required"}
        return tools.fetch_build_status(build_id)


def create_fastmcp_server():
    from mcp.server.fastmcp import FastMCP

    mcp = FastMCP("stack-sentinel-mcp")
    register_tools(mcp)
    return mcp
```
